## Goal

Provide human-friendly name resolution across the WireGuard mesh so the operator can `ssh mom.wg` instead of remembering `10.100.0.10`. DNS is served by CoreDNS on the VPS hub, and the zone is generated from the same network state file that drives everything else.

## Success criteria

- All peers are resolvable by `<name>.wg` from any node on the WireGuard network.
- DNS zone is auto-generated by the provisioning CLI from network state.
- Adding or removing a peer updates the zone and reloads CoreDNS as part of `wgmesh sync`.
- Client WireGuard configs set DNS to the hub's tunnel IP so resolution works when the tunnel is up.

## Scope boundaries

**In scope:**

- CoreDNS with file plugin serving a `.wg` zone on the hub's WireGuard interface IP
- Jinja2 template for the zone file, rendered from `network.sops.yaml`
- Forward and reverse DNS records for all peers
- Client WireGuard configs include `DNS = <hub_wg_ip>` and `PostUp`/`PostDown` hooks for `/etc/resolv.conf` management if needed
- Zone regeneration and CoreDNS reload as part of `wgmesh sync`

**Out of scope:**

- Split-horizon DNS or conditional forwarding for non-`.wg` queries
- DNSSEC
- Dynamic DNS updates (zone is statically generated from state)
- mDNS or Avahi integration
