# ADR-006: Hub Provisioning via Terraform and Ansible

## Context

The Porthole hub must be treated as ephemeral infrastructure (see SPIKE-007):
capable of being destroyed and rebuilt from repo state at any time, with no
manual steps and no data loss (all persistent state lives in `network.sops.yaml`).

The current implementation (`wgmesh bootstrap`) provisions the hub by SSHing
into a VPS and running inline shell scripts generated by the wgmesh CLI. This
approach:

- Is not idempotent — re-running risks duplicating configuration.
- Is not declarative — desired state is spread across Python source, not IaC.
- Does not manage DNS records or cloud firewall rules.
- Cannot rebuild the hub from scratch without manual VPS creation first.
- Conflates network state management (wgmesh CLI's job) with infrastructure
  provisioning (IaC's job).

As the project adds a node bootstrap TUI (SPEC-009) that can trigger hub
spin-up on demand, the provisioning layer must be scriptable and fully
automated.

## Decision

Use **Terraform** for hub VPS lifecycle (provision, destroy, DNS records, cloud
firewall) and **Ansible** for hub configuration (WireGuard, CoreDNS, nftables,
Guacamole, Caddy, tunnel user).

- Terraform manages cloud resources. The reference provider is DigitalOcean;
  the Ansible layer is provider-agnostic.
- Ansible reads `network.sops.yaml` (decrypted via SOPS at runtime) to
  populate WireGuard peer configs, CoreDNS zone, and tunnel `authorized_keys`.
  This replaces the per-role Jinja2 rendering currently done by the wgmesh CLI
  at bootstrap time.
- The `wgmesh bootstrap` command is superseded. It remains in place during the
  transition and is a candidate for deprecation once SPEC-008 is validated.

## Alternatives Considered

| Alternative | Reason not chosen |
|-------------|-------------------|
| Keep `wgmesh bootstrap` (bash-via-SSH) | Not idempotent; can't rebuild from scratch; mixes concerns |
| Pulumi | More capable but significantly more complex for a single-operator fleet; no clear advantage |
| NixOS / Nix flakes | Elegant but requires the operator to adopt Nix, which is a large dependency not justified by fleet size |
| Cloud-init only | No ongoing configuration management; can't re-apply state changes without rebuilding the VPS |
| Chef / Puppet | Heavyweight; no existing operator familiarity; overkill for ~1 server |

## Consequences

**Positive:**
- Hub is fully rebuildable from `terraform apply && ansible-playbook site.yml`
  with no manual steps.
- Infrastructure state is version-controlled alongside network state.
- Ansible roles are individually testable and reusable.
- Aligns with the 202602-workstation project's Ansible conventions, reducing
  cognitive overhead for the operator.

**Accepted trade-offs:**
- Terraform state must be stored and managed (local by default; remote backend
  for production hardening).
- Operator must have Terraform and Ansible installed — handled by the node
  bootstrap TUI (SPEC-009) on the operator's machine.
- Cloud provider credentials must be available in the environment during hub
  spin-up.
- The Terraform provider choice (DigitalOcean as reference) is not portable
  across providers without additional work, though the Ansible layer is
  provider-agnostic.
