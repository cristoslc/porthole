# ADR-007: Age-encrypted setup key and cloud-init WireGuard bootstrap

## Context

ADR-006 adopted Terraform + Ansible for hub provisioning. The current flow requires public SSH access to the hub for the Ansible post-provisioning step. This creates two problems:

1. **SSH exposure.** The hub VPS has port 22 open to the internet during and after provisioning. Even with key-only auth, this is unnecessary attack surface for a VPS whose only public-facing service should be WireGuard (UDP 51820).

2. **Credential conflation.** The SSH key used for provisioning is the operator's personal key, which is also (or could be) used for guacamole remote desktop tunnels. A single compromised credential grants both infrastructure management and desktop access.

SPIKE-001 investigated alternatives including Pangolin, Headscale, Netmaker, and other coordination servers. All require additional infrastructure or ecosystem lock-in. The spoke N+1 problem (enrolling new peers after hub creation) means SSH cannot be fully eliminated without a coordination server.

## Decision

**Introduce a setup-only SSH key encrypted with age, and use cloud-init to bootstrap WireGuard at hub first boot.**

### Three-credential separation

| Domain | Credential | Storage | Scope |
|--------|-----------|---------|-------|
| WireGuard tunnel | Hub + peer WG keypairs | `network.sops.yaml` (age) | Kernel WireGuard, all peers |
| Hub provisioning | Setup SSH ed25519 key + passphrase | `setup-key.sops.yaml` (age) | `porthole` user on hub, limited sudo |
| Remote desktop | Per-peer passwords/keys | `network.sops.yaml` (age) | Guacamole server only |

The setup key:
- Is generated by `porthole init` alongside the WireGuard keypairs.
- Has a passphrase as a second factor.
- Is encrypted with age and committed to the repo as `setup-key.sops.yaml`.
- Authenticates as a dedicated `porthole` user with sudo limited to `wg`, `wg-quick`, `systemctl`.
- Is rotatable and revocable independently of all other credentials.

### Cloud-init WireGuard bootstrap

Terraform injects cloud-init user-data that:
1. Configures WireGuard with the hub's private key and known peer public keys.
2. Creates the `porthole` user with only the setup SSH public key.
3. Enables a firewall: WireGuard (51820/udp) open, SSH restricted to the WG subnet (10.100.0.0/24) after a brief provisioning window.

The hub boots with WireGuard already running. The first spoke connects immediately. Public SSH auto-closes.

### Spoke N+1 enrollment

From a workstation already on the WireGuard network:
```
porthole add spoke-name
porthole sync    # SSH to hub@10.100.0.1 over WG tunnel, using setup key
porthole gen-peer-scripts spoke-name
```

From a workstation not yet on the network (e.g., fresh re-provisioning):
```
porthole hub-ssh-open   # temporarily re-opens public SSH on the hub
porthole sync           # SSH to hub@public-ip using setup key
porthole hub-ssh-close  # closes public SSH again
```

## Alternatives Considered

| Alternative | Reason not chosen |
|-------------|-------------------|
| Eliminate SSH entirely (cloud-init only) | Spoke N+1 enrollment requires updating hub's WG peer list; no management channel without SSH or a coordination server |
| Headscale / Netmaker coordination server | Additional infrastructure to run, vendor lock-in to their client ecosystems; overkill for hub-and-spoke |
| HTTPS enrollment API on hub | Custom code to build and maintain; larger attack surface than SSH |
| Nebula (certificate-based overlay) | Not WireGuard; would require abandoning the existing architecture |
| Keep personal SSH key for everything | Conflates provisioning and desktop access; no credential separation |

## Consequences

**Positive:**
- Hub's only permanently public port is WireGuard (51820/udp). SSH is WG-only by default.
- Credential domains are separated — compromising one doesn't grant access to others.
- Setup key is portable (encrypted in repo) — any authorized operator can provision from any workstation.
- Uses only boring, proven technology: SSH, age/sops, cloud-init, ufw. No new dependencies.
- Cloud-init bootstrap means hub is WireGuard-reachable at first boot, before any Ansible runs.

**Accepted trade-offs:**
- Ansible still runs over SSH, just over the WireGuard tunnel instead of public internet. This is strictly better, not a new limitation.
- The setup key's private key is in the repo (encrypted). Repo access + age key + passphrase = hub management access. This is acceptable for a single-operator fleet.
- The brief public SSH window at first provision (or `hub-ssh-open` for re-enrollment) is a small exposure. Mitigated by key-only auth and short duration.
- cloud-init WireGuard support varies slightly across providers. Both target providers (Hetzner, DigitalOcean) support it.
